# Релизы — как выпустить новую версию приложения

## Как это работает

Каждый раз, когда ты создаёшь **git-тег** вида `v1.0.0` и пушишь его на GitHub,
автоматически запускается сборка. Через несколько минут в разделе
**Releases** на GitHub появится новый релиз с готовым APK-файлом.

```
Ты создаёшь тег v1.2.0
        │
        ▼
GitHub Actions запускает сборку (ubuntu, JDK 17)
        │
        ▼
Gradle собирает Release APK
        │
        ▼
APK прикрепляется к релизу на GitHub
        │
        ▼
Любой может скачать и установить medicine-tracker-v1.2.0.apk
```

---

## Быстрый старт — выпустить первый релиз

```bash
# 1. Убедись, что все изменения закоммичены
git add .
git commit -m "Готов к релизу v1.0.0"

# 2. Создай тег
git tag v1.0.0

# 3. Отправь тег на GitHub
git push origin v1.0.0
```

Готово! Открой вкладку **Actions** в репозитории и наблюдай за сборкой.

## Правила именования версий (Semantic Versioning)

Версии называются по схеме `vМАЖОР.МИНОР.ПАТЧ`:

| Изменение | Что увеличить | Пример |
|---|---|---|
| Сломали совместимость, переделали всё | МАЖОР | `v1.0.0 → v2.0.0` |
| Добавили новую функцию (ничего не сломали) | МИНОР | `v1.0.0 → v1.1.0` |
| Исправили баг | ПАТЧ | `v1.0.0 → v1.0.1` |

## Как версия попадает в APK

Тег `v1.2.3` автоматически становится версией приложения:

| Параметр | Значение | Где видно |
|---|---|---|
| `versionName` | `1.2.3` | В настройках телефона → «О приложении» |
| `versionCode` | кол-во коммитов | Google Play (монотонно растёт) |

В `build.gradle.kts` это настроено так:
```kotlin
versionCode = System.getenv("VERSION_CODE")?.toIntOrNull() ?: 1
versionName = System.getenv("VERSION_NAME") ?: "1.0"
```

## Подпись APK

### Сейчас (debug-ключ)

По умолчанию APK подписывается **debug-ключом** — тем же, что используется
при разработке. Такой APK:
- ✅ Устанавливается на любой Android-телефон
- ✅ Работает без ограничений
- ❌ Нельзя публиковать в Google Play
- ❌ Нельзя обновить поверх APK с другой подписью

### Настройка release-подписи (для серьёзного использования)

Чтобы использовать собственный ключ подписи, нужно:

**Шаг 1 — Создай keystore (один раз):**
```bash
keytool -genkey -v \
  -keystore my-release-key.jks \
  -alias my-key-alias \
  -keyalg RSA -keysize 2048 \
  -validity 10000
```
Команда задаст несколько вопросов (имя, организация и т.д.) и создаст файл `my-release-key.jks`.

**Шаг 2 — Закодируй keystore в base64:**
```bash
# Linux / Mac
base64 -i my-release-key.jks

# Windows (PowerShell)
[Convert]::ToBase64String([IO.File]::ReadAllBytes("my-release-key.jks"))
```
Скопируй полученную строку.

**Шаг 3 — Добавь секреты в GitHub:**

Открой репозиторий на GitHub → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**

| Имя секрета | Значение |
|---|---|
| `KEYSTORE_BASE64` | строка base64 из шага 2 |
| `KEYSTORE_PASSWORD` | пароль от keystore |
| `KEY_ALIAS` | псевдоним ключа (например `my-key-alias`) |
| `KEY_PASSWORD` | пароль от ключа |

После добавления секретов следующий релиз будет подписан твоим ключом.

## Часто задаваемые вопросы

**Q: Как удалить тег, если я ошибся?**
```bash
git tag -d v1.0.0               # удалить локально
git push origin --delete v1.0.0 # удалить на GitHub
```

**Q: Можно ли собрать APK без создания релиза?**

Да — на вкладке **Actions** в GitHub выбери нужный запуск и скачай
APK из раздела **Artifacts** (если добавить шаг `upload-artifact` в workflow).

**Q: Сколько времени занимает сборка?**

Обычно 5–10 минут: скачивание зависимостей, компиляция, сборка APK.
Повторные сборки быстрее благодаря кэшу Gradle.

**Q: Как посмотреть логи сборки?**

GitHub → вкладка **Actions** → выбери нужный запуск → нажми на шаг чтобы развернуть логи.

**Q: Где скачать готовый APK?**

GitHub → вкладка **Releases** → нажми на нужный релиз → раздел **Assets**.
